Physical Persistence Schema

MongoDB stores source evidence, derived analytical records, runtime state, and retrieval chunks; Neo4j and ZVec complement it with graph topology and vector search.

retrieval index
domain and runtime collections
documents
source collection and pipeline checkpoint state
  • PKid unique sparse
  • title, content, source_url
  • source, source_name, published_date
  • graph_extracted, structured_extracted
  • chunks_created, quality_score
entity_mentions
one entity occurrence per document
  • IXentity_id
  • FKdocument_id
  • entity_name, entity_type
  • source_name, published_date
  • created_at
relation_assertions
relation evidence mirrored from graph extraction
  • IXfrom_entity_id, to_entity_id
  • relation_type, confidence
  • evidence, status
  • source_doc_ids, created_at
document_chunks
semantic retrieval units stored in MongoDB
  • PKchunk_id unique sparse
  • FKdocument_id
  • chunk_text, enriched_text
  • chunk_index, section_header
  • entity_ids, source_name
ZVec vectors
external vector index, not a MongoDB collection
  • PKchunk_id
  • embedding, content
  • metadata.document_id
  • metadata.entity_ids
  • metadata.published_date
structured records
MongoDB analytical collections from extraction step 2
  • bills: id, status, chamber
  • votes: id, bill_id, date
  • committees: id, chamber
  • contributions: donor_id, recipient_id
  • conflict_flags: severity, created_at
runtime collections
application state and audit records
  • users: id, email, is_admin
  • chat_sessions: id, updated_at
  • chat_messages: session_id, timestamp
  • query_history: created_at

The schema is intentionally document-centered: every derived collection keeps provenance through document, chunk, session, or entity identifiers, while specialized stores handle graph traversal and vector similarity.
